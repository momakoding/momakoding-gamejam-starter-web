## 这是个什么东西？

这是一个**"开箱即用的网页小游戏脚手架"**。

你可以把它想象成 PPT 的空白模板：PPT 模板已经帮你排好版式、配好颜色、留好占位图，你只要往里面填内容就行。

这个项目做了类似的事情，只不过填进去的不是文字和图片，而是**游戏的玩法**：怎么跳、怎么打怪、怎么得分、怎么过关。

开箱自带：

- 一个**首页菜单**（开始游戏、玩法介绍、关于我们）
- 一个**能跑的示例小游戏**（控制一个方块跳来跳去收集星星）
- 一套**暂停 / 继续 / 退出**的通用逻辑
- **不需要任何美术素材**也能先跑起来（程序会自动画出彩色方块当占位）
- 能直接在浏览器里玩，也能打包发给别人

换句话说，你拿到它的那一刻，已经有一个**能玩的游戏**了。你要做的是把它**改成你的游戏**。

## 它适合做什么样的游戏？

适合做：

- **2D** 的横版 / 俯视角小游戏（平台跳跃、躲避、收集、打砖块、扫雷、推箱子……）
- **一局几分钟**的轻量玩法，适合 Game Jam、课堂作业、产品 Demo、面试作品
- **在网页里玩**的游戏（手机浏览器也能跑）
- 想快速验证"**这个点子好不好玩**"的原型

不太适合：

- 3D 大作（那是 Unity / Unreal 的地盘）
- 需要联网对战、强后端服务的多人游戏
- 画面极其精致的商业级产品

一句话：**"一个人一周能做完的小游戏"**，这套东西就是为它量身定做的。

## 为什么产品经理会喜欢它？

因为它把"**从创意到能玩**"这条路**压到最短**了。

你通常遇到的困难是：

1. "我有个点子，但我不会写代码。"
2. "找程序员实现，来回讲需求要一周，做出来又不是我想要的。"
3. "原型工具（Figma、Axure）只能点点点，**没法真的玩**，拿给老板看他 get 不到。"

这个项目配合现在的 **AI 编程助手**（比如 Claude Code、Cursor、Kilo Code 等），能让你：

- **用中文描述你想要的效果**，让 AI 帮你改代码
- 改完**立刻在浏览器里玩到**，不满意就继续说"跳得再高一点"、"星星再密一点"、"加个计时器"
- **不需要懂代码长什么样**，你只需要会**描述体验**

而这正是产品经理的本职工作：**用人话把"我想要什么感觉"讲清楚**。

## 一个真实的工作流长什么样？

假设你脑子里有个点子：**"一个厨师躲避从天上掉下来的菜刀，接住掉下来的食材做菜"**。

传统流程是这样：

```
你 → 写需求文档 → 约程序员开会 → 程序员估工期
  → 一周后给你 Demo → 你说"不是这个感觉" → 再改一周……
```

用这个脚手架 + AI 助手，是这样：

```
第 1 天上午
  你: "把示例里的方块换成一个厨师，星星换成食材，
       加一些从上面掉下来的菜刀，被砍到就 Game Over"
  AI: (改代码)
  你: (刷新浏览器，玩 30 秒)
      "菜刀掉得太慢了，再快一倍；食材颜色改成红色"
  AI: (改代码)
  你: (再玩 30 秒) "可以，下一步加计时器"

第 1 天下午
  你把能跑的 Demo 甩群里: "看看这个，好玩吗？"
```

**你没写一行代码，但你拿到了一个能玩的东西。**

这个脚手架刻意把项目结构、命名规则、数值参数都**整理得很规整**，就是为了让 AI 改起来不出错、改完还跑得起来。

## 我需要准备什么？

理想情况下你需要一台电脑，加下面三样东西。**都是免费的**。

### 1. Node.js（电脑运行网页项目的基础环境）

去 [nodejs.org](https://nodejs.org) 下一个 **LTS 版**装上。一路点"下一步"。

### 2. pnpm（安装项目依赖用的工具）

装完 Node.js 后，打开"终端"（Windows 叫"命令提示符"或"PowerShell"，Mac 叫"终端"），粘贴：

```bash
npm install -g pnpm
```

回车，等它跑完。

### 3. 一个 AI 编程助手

推荐任选其一：

- **[Kilo Code](https://kilocode.ai/)**（本项目就是用它在开发）
- **[Cursor](https://cursor.sh/)**（一个带 AI 的编辑器，上手最平滑）
- **[Claude Code](https://claude.com/claude-code)**（命令行里直接和 Claude 对话改代码）

这些工具都能**读懂这个项目的结构**，然后按你说的话改代码。

## 三步跑起来

打开终端，进到这个项目的文件夹，依次跑：

```bash
# 第一步：把需要的零件都装好（第一次才要，之后不用再跑）
pnpm install

# 第二步：启动
pnpm dev
```

终端会打印一个地址，大概长这样：

```
  ➜  Local:   http://localhost:5173/
```

把这个地址**复制到浏览器**，你就看到游戏首页了。点"开始游戏"进 Demo，按方向键移动、上键跳跃、ESC 暂停。

> 想停掉？回到终端按 `Ctrl + C`。

要**打包发给别人**？

```bash
pnpm build
```

会在 `dist/` 文件夹里生成一堆文件，整个文件夹丢到任何静态网站托管（GitHub Pages、Vercel、Netlify……）就能让别人访问。

## 跑起来之后，先改这几处占位内容

Demo 跑起来后，你会看到首页标题、玩法介绍、关于我们页面里有一些**占位文字**（比如 `[请输入文本]`、`[团队名称]`、`[成员 A]`）。这些都是留给你填的。

直接告诉 AI：

> "帮我把 `game-info` 里的占位内容替换成我的游戏信息：游戏叫 XX，副标题是 XX，团队叫 XX，成员有 A、B、C。"

或者自己打开这几个文件改：

| 要改什么 | 文件 |
|---|---|
| 游戏名称、副标题 | `src/contents/game-info/game-meta.ts` |
| 玩法介绍正文 | `src/contents/game-info/how-to-play.md` |
| 团队名称、成员 | `src/contents/game-info/team.ts` |
| 整体配色 | `src/style.css` 里的 `@theme { ... }` |

改完刷新浏览器，首页就变成你自己的游戏了。

## 把你的游戏创意记录下来

有了想法之后，直接告诉 AI 你想做什么游戏。AI 会把你的描述整理成一份**游戏设计 spec**，保存在 [`docs/vibe/spec-game.md`](../vibe/spec-game.md)。

这份 spec 就是你的游戏的"设计蓝图"，包括：核心玩法、玩家操作、实体（角色/敌人/道具）、胜负条件、计分规则、难度曲线等。

**怎么用？**

1. 跟 AI 说："我想做一个 XXX 游戏"（用中文就行，随便怎么描述）
2. AI 会自动生成 `docs/vibe/spec-game.md`
3. 后续你想改玩法、加机制，AI 会**更新**这份 spec 而不是从头写
4. 所有开发工作都以这份 spec 为准，保证你的创意不会在来回修改中走样

> 💡 你不需要自己打开这个文件编辑。只要跟 AI 聊你的想法，它会帮你维护这份文档。

## 怎么"指挥" AI 改游戏？

核心心法只有一条：**把你想要的"感觉"说清楚，而不是指挥"怎么改代码"**。

### 推荐这样说

- "跳起来感觉太重了，想要**漂浮一点**的感觉"
- "主角从蓝色方块换成一只**像素风的猫**"（AI 会帮你生成占位图，或指导你去哪找素材）
- "加一个**倒计时 60 秒**，时间到了就显示 Game Over"
- "星星太好吃了，**从 10 个减到 5 个**，而且颜色闪一闪再消失"
- "背景**从黑色换成渐变的夕阳色**"
- "加一个**主菜单的 BGM**，用[某某音效网站]里的 xxx.mp3"

### 尽量别这样说

- "把 `game-scene.ts` 里的 `GRAVITY` 改成 400"（除非你已经懂了，不然让 AI 自己决定改什么参数）
- "重构一下整个状态管理"（Game Jam 阶段不需要，**能跑 > 优雅**）
- "加一个服务器后端存档功能"（超纲了，小游戏用不到）

### 一些有用的"套路词"

跟 AI 说话时，这些词能让它更懂你：

| 你想表达                       | 对 AI 的说法                         |
| ------------------------------ | ------------------------------------ |
| 操作手感/物理感觉              | "feel"、"手感"、"跳跃曲线"、"惯性"   |
| 画面效果                       | "调色"、"粒子"、"震屏"、"闪白"       |
| 游戏规则变化                   | "加一个机制"、"胜利条件改成……"       |
| 界面                           | "HUD"（游戏里那些分数、血条的统称）  |
| "我不要你改架构，就想调个数字" | "只改数值，不动结构"                 |

## 我最容易搞砸的地方？

### 1. 一次要求太多

**别说**："把主角换成猫，加三种敌人，做三个关卡，加一个商店系统，再做个排行榜。"

**要说**：先做猫，跑起来看看；OK 了再加敌人；敌人 OK 了再加关卡……

AI 一次做太多事情，出错率会指数级上升。**小步快跑，随时能玩**，这才是 Game Jam 节奏。

### 2. 不玩就继续加需求

每次 AI 改完，**打开浏览器玩 30 秒**再说下一句话。你的下一句话应该基于你刚才的手感，而不是脑子里原本的计划。

### 3. 出了 bug 就慌

浏览器白屏、终端飙红字，都是家常便饭。把**终端里的红字**或者**浏览器 F12 控制台的红字**原样贴给 AI，大部分问题它自己就修了。

### 4. 想让画面像 3A 大作

占位图 + 像素风 + 简单配色，就能做出**很有味道**的小游戏。把精力花在玩法上，美术可以最后再找朋友 / AI 生成 / 买素材补。

## 常见问题

**Q：我完全不懂代码，真的能做出东西吗？**

能。本项目就是为此设计的。你需要的是**清楚表达想要什么**的能力，和**不怕试错**的心态。

**Q：AI 会把项目改坏吗？**

可能会。所以每次重大改动前，学一个小技能：**Git 提交**（commit）。用 VS Code / Cursor 自带的图形界面点一下按钮就行。改坏了可以一键回到上一个能跑的版本。让 AI 教你用就行。

**Q：免费吗？**

脚手架本身完全免费开源。Node.js、pnpm、编辑器都免费。AI 助手大部分有免费额度，重度使用大概每月 20 美金左右。

**Q：做出来的游戏能卖钱 / 参赛 / 商用吗？**

这个脚手架是开源的，可以自由商用。你做出来的游戏归你自己。但记得**给你用到的素材（音乐、图片、字体）检查授权**。

**Q：能做手机 App 吗？**

能在**手机浏览器**里玩。想打包成真正的 App（App Store / Google Play），需要另外一层包装（Capacitor、Tauri 之类），可以做，但属于下一步的话题。

**Q：我学了一半不想学了怎么办？**

没事，你至少已经：

- 跑通了一个前端项目（很多人一辈子没跑通过）
- 和 AI 协作编程了一次（这是 2026 年的核心技能）
- 有了一个能玩的小游戏（哪怕是 Demo）

这些对 PM 的简历来说都不亏。
